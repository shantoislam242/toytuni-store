# Backend Phase 2: Auth Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Real authentication (Supabase Auth — email+password AND Google OAuth), an email-allowlist admin gate on `/admin/*`, and customer accounts, replacing every `isLoggedIn` mock.

**Architecture:** `@supabase/ssr` session via middleware + an auth callback route. A pure `isAdmin(email)` allowlist helper gates admin. Server helpers + a client `useAuth()` expose the session. Customer rows link to the auth user; the account page shows the user + order history.

**Tech Stack:** Next.js 16 (App Router), TypeScript, Supabase Auth (`@supabase/ssr`, `@supabase/supabase-js`), Vitest.

**Spec:** `docs/superpowers/specs/2026-07-17-auth-phase2-design.md`

## Global Constraints

- **Non-standard Next.js.** Before writing `middleware.ts`, the callback route handler, or any `cookies()`/session code, READ `node_modules/next/dist/docs/01-app/02-guides/authentication.md` and the relevant middleware / route-handler / cookies references. Do not write these from memory.
- **Two auth methods:** email+password and Google OAuth, both wired into the existing `/signin` UI (which already has both forms).
- **Admin = email in `ADMIN_EMAILS`** (comma-separated env var), matched case-insensitively. Never hardcode an admin email in source.
- **`SUPABASE_SERVICE_ROLE_KEY` is server-only.** Customer-link and order-history reads use it server-side; never in client code.
- **Reuse Phase 1 clients** (`src/lib/supabase/{client,server,admin}.ts`) — do not create parallel Supabase clients; extend if needed.
- `.env.local` / `.superpowers/` gitignored — stage explicit paths only, never `git add -A`.
- BDT integer prices unchanged; auth must not alter Phase 1 data behavior.

## Task 1: `isAdmin` allowlist helper (TDD)

**Files:** Create `src/lib/auth/admin.ts`, `src/lib/auth/admin.test.ts`.

**Interfaces:** `export function isAdmin(email: string | null | undefined): boolean` — true iff `email` (trimmed, lowercased) is in `process.env.ADMIN_EMAILS` (comma-separated, each trimmed/lowercased).

- [ ] **Step 1 — failing test** `src/lib/auth/admin.test.ts`:
```ts
import { describe, it, expect, beforeEach } from "vitest";
import { isAdmin } from "@/lib/auth/admin";

beforeEach(() => { process.env.ADMIN_EMAILS = "boss@toytuni.com, Admin@Shop.com"; });

describe("isAdmin", () => {
  it("matches an allowlisted email", () => expect(isAdmin("boss@toytuni.com")).toBe(true));
  it("is case-insensitive and trims", () => expect(isAdmin("  ADMIN@shop.com ")).toBe(true));
  it("rejects a non-admin", () => expect(isAdmin("someone@else.com")).toBe(false));
  it("rejects null / empty", () => { expect(isAdmin(null)).toBe(false); expect(isAdmin("")).toBe(false); });
  it("false when ADMIN_EMAILS unset", () => { delete process.env.ADMIN_EMAILS; expect(isAdmin("boss@toytuni.com")).toBe(false); });
});
```
- [ ] **Step 2 — run `npm test -- auth/admin`, confirm RED.**
- [ ] **Step 3 — implement** `src/lib/auth/admin.ts`:
```ts
/** True iff the email is in the ADMIN_EMAILS allowlist (comma-separated,
 *  case-insensitive). The single source of admin truth. */
export function isAdmin(email: string | null | undefined): boolean {
  if (!email) return false;
  const list = (process.env.ADMIN_EMAILS ?? "")
    .split(",").map((e) => e.trim().toLowerCase()).filter(Boolean);
  return list.includes(email.trim().toLowerCase());
}
```
- [ ] **Step 4 — run test, confirm GREEN (5/5).**
- [ ] **Step 5 — commit** `git add src/lib/auth/admin.ts src/lib/auth/admin.test.ts && git commit -m "feat(auth): admin email-allowlist helper"`.
